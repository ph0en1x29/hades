# Dataset Adequacy Strategy

*Responding to Dr. Liu's directive: "Before you can convince me and other experts that your datasets are adequate, your proposed approach/prototype cannot be validated in a scientific way."*

## The Core Problem

CIC-IDS2017/2018 provides **network flow features** (packet counts, durations, byte ratios) with binary labels (Benign/Attack). A real SOC analyst never sees these features. They see:

```
[2026-03-12 10:03:41] ALERT: Sigma Rule SID-4625 fired
  Rule: "Multiple Failed Logons from Single Source"
  Source: 198.51.100.50 → 10.0.0.15:445
  MITRE: T1110.001 (Password Guessing)
  Severity: HIGH
  Context: 47 failed logon events in 120s, 3 distinct accounts targeted
```

The gap: CICIDS has the traffic that *would* generate this alert, but not the alert itself, not the rule that fired, not the MITRE mapping, and not the analyst context. Without these, any evaluation is testing "can an LLM classify flow features" — which is a ML problem, not a SOC triage problem.

## What "Scientifically Adequate" Requires

A dataset is adequate for SIEM triage research when it satisfies **all five** criteria:

| # | Criterion | Why | CICIDS | Splunk Attack Data | SOC-Bench |
|---|-----------|-----|--------|-------------------|-----------|
| C1 | Detection rule association | Analysts triage alerts, not packets | ❌ | ✅ | ✅ |
| C2 | MITRE ATT&CK mapping | Standard taxonomy for technique coverage | ❌ | ✅ | ✅ |
| C3 | Alert-level context | Severity, message, source/dest, timestamps | ⚠️ partial | ✅ | ✅ |
| C4 | Label provenance | How ground truth was determined | ⚠️ flow labels | ✅ rule-linked | ✅ scenario-based |
| C5 | Multi-stage correlation | Real attacks span multiple alerts | ❌ individual flows | ⚠️ per-technique | ✅ full campaigns |

**CICIDS satisfies 0/5 fully.** This is why Dr. Liu flagged it.

## Three-Phase Dataset Strategy

### Phase 1: Public Benchmark (Splunk Attack Data) — NOW

**What:** Curated attack telemetry from Splunk's Attack Range simulations, each paired with the exact Splunk detection rule that fires on it.

**Why it's adequate:**
- Every dataset has a matching detection rule (C1 ✅)
- Rules are mapped to MITRE ATT&CK techniques (C2 ✅)
- Raw Sysmon/Windows Security logs provide full alert context (C3 ✅)
- Labels derived from detection rules, not manual annotation (C4 ✅)
- Can construct multi-technique scenarios from same Attack Range runs (C5 ⚠️)

**What we build:**
1. Download Splunk Attack Data for selected techniques (credential access, discovery, lateral movement, execution, persistence)
2. Download matching Splunk Security Content detection rules
3. Parse telemetry through detection rules → generate alert records
4. Normalize to UnifiedAlert with full benchmark context
5. Document provenance chain: raw log → rule match → alert → normalized

**Deliverable:** 500-1,000 benchmark alerts across 5+ MITRE tactics, each with rule association, provenance, and scenario grouping.

**Adequacy argument:** "Every alert in our benchmark was generated by applying a published detection rule to real attack telemetry. The rule-to-technique mapping is maintained by Splunk's security research team and covers 400+ MITRE techniques."

### Phase 2: Campaign Benchmark (SOC-Bench) — PENDING DR. LIU

**What:** Dr. Liu's Colonial Pipeline reconstruction — multi-stage ransomware campaign with coordinated telemetry across EDR, SIEM, network, and file-system sources.

**Why it's stronger:**
- Full kill-chain campaigns, not isolated techniques (C5 ✅✅)
- Ground truth from reconstructed incident, not synthetic labels
- Formal scoring rubric (ring-based, penalty-aware)
- Direct alignment with advisor's research program

**Action:** Ask Dr. Liu in first meeting whether Hades can be evaluated as a SOC-Bench system.

### Phase 3: Adversarial Extension — AFTER PHASES 1-2

Only after baseline dataset adequacy is established:
- Generate adversarial variants from Phase 1/2 alerts
- Inject payloads into realistic log fields
- Measure attack success rate against validated baseline

**The key insight for Dr. Liu:** We don't evaluate adversarial robustness until we can prove the baseline triage is scientifically valid. The dataset gate enforces this ordering.

- **Publicly downloadable** (Splunk repos are open-source)
- **Version-pinned** (manifest records exact commit SHAs)
- **Transform-documented** (parser version + projection version tracked per alert)
